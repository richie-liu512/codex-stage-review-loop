# Codex Stage Review Loop

[English](README.md) | 简体中文

这是一个 Codex Skill，用来把“先检查一下再继续”变成一个轻量、可复用的任务验收流程。它会让 Codex 回到用户的原始要求，检查真实证据，运行可用验证，修复已确认的阻断问题，并在需要用户判断时停下来。

它不承诺让模型靠自我反省发现所有错误。它的实际价值更具体：让任务收尾不再只靠一句“看起来完成了”，而是有范围、有证据、有停止条件。

## 为什么需要它

AI 编程助手经常能完成主要改动，但在最后一步出问题：漏跑验证、把未经确认的判断当结论、把可选建议当成必须改动，或者在证据不足时给出过强的完成声明。

这个 Skill 解决的是这类“最后一公里”的验收问题。它适合用于代码改动、文档修改、生成物检查、本地配置调整，或者另一个 Codex Skill 修改完成之后。

## 它怎么工作

- 先锁定审查范围：用户要求、改动文件、验收证据和明确排除项。
- 检查每个问题、修复和建议是否仍服务于用户的原始目标，避免为了流程完整而制造额外工作。
- 优先检查真实证据，例如 diff、文件、测试、构建结果、日志、渲染结果或冒烟检查。
- 再做一次新视角审查，寻找漏掉的需求、二次 Bug 和用户可见问题。
- 只在问题已被证据确认、且修复范围低风险时自动修复。
- 把可选优化作为建议报告，不擅自扩大任务。
- 遇到需求歧义、风险操作或产品取舍时停止并询问用户。

## 两种模式

### 轻量模式

默认模式，适合日常频繁使用。它会在当前会话里做一次紧凑的验收检查和一次新视角审查，并带有小范围修复循环。

示例：

```text
使用 $stage-review-loop 轻量验收当前工作，并修复真实阻断问题。
```

### 独立审查者模式

更强的模式，适合发布前确认或风险较高的改动。可用时，它会优先使用独立 reviewer、新线程、新模型上下文或其他真正独立的审查表面。

如果没有真正独立的审查表面，它应该明确说明自己只是“在当前上下文内模拟新视角检查”，而不是完整独立审查。

示例：

```text
使用 $stage-review-loop 独立审查者模式，检查结果是否符合原始需求。
```

## 和 Codex 官方审查的关系

它不是 Codex 内置代码审查或 PR 审查的替代品。代码 diff 级别的问题，仍然更适合交给官方 review 能力。

这个 Skill 关注的是更上层的问题：这个任务是否真的符合用户要求，验证是否实际发生，是否还有阻断问题没有处理。

两者可以串联使用：先用代码审查找 diff 里的具体问题，再用这个 Skill 做任务级验收。

## 边界

这个 Skill 可以减少漏验收、范围漂移和无证据收尾，但不能可靠发现所有问题。需要外部事实、领域知识、测试覆盖、明确需求或真正独立审查者的问题，不能只靠它解决。

它刻意保持轻量。除非用户明确要求更强的最终审查，否则不应该变成大型发布流程。

## 安装

把 Skill 文件夹复制到 Codex skills 目录：

```text
stage-review-loop/
  SKILL.md
  agents/
    openai.yaml
```

然后按名称调用：

```text
使用 $stage-review-loop 验收当前工作。
```
