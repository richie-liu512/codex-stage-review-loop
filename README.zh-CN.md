# Codex Stage Review Loop

[English](README.md) | 简体中文

这是一个为 Codex 设计的轻量检查 Skill，用来在执行下一步前“把把关”。

它的逻辑很直接：回头看你的原始需求，拿真实的结果（比如代码差异、文件状态、测试结果、日志或报错）来验证。发现安全可控的小问题就顺手修了，但拿不准、有风险的地方还是要停下来问你。

它的核心目的不是去盲猜所有未知的 Bug，而是为 AI 划定一条清晰的验收底线：防止模型在收尾时凭感觉敷衍交差，确保每一次“任务完成”都清清楚楚、有理有据。

## 为什么需要它

AI 编程助手通常能完成主要任务，却常在“最后一公里”翻车：例如遗漏测试验证、将未经测试的推测当作结论、将可选建议强行合入，或在缺乏验证结果的情况下盲目宣布任务完成。

本 Skill 专为解决此类验收问题而生。它不仅适用于代码变更，也同样服务于文档更新、生成物检查、本地配置调整，或作为其他 Codex Skill 执行完毕后的收尾环节。

## 它怎么工作

- **锁定审查范围**：明确用户的原始需求、变更的文件、验收标准以及明确排除的项。
- **目标对齐**：确保提出的问题、修复方案和建议都严格服务于原始目标，避免为了“形式上的完整”而过度优化或制造额外工作量。
- **证据导向**：优先核查可检查证据和可用验证结果，例如 diff 差异、文件状态、测试报告、构建日志、渲染效果或冒烟检查。
- **新视角复查**：以重新审视的方式排查遗漏的需求、引入的新 Bug 以及用户侧可见的异常。
- **克制修复**：仅当问题被可检查证据或验证结果证实，且修复操作位于当前范围内、风险较低时，才自动执行修复。
- **建议分离**：将非必要的优化作为“可选建议”提出，绝不擅自扩大修改范围。
- **适时熔断**：在遇到需求存在歧义、高风险操作、凭证、删除、配置调整，或需要进行产品级取舍时，主动停止并请求用户介入。

## 两种模式

### 轻量模式（默认）

适合高频的日常使用场景。它会在当前会话中执行一次紧凑的验收与复查，并包含小范围的自动修复循环。

示例指令：

```text
使用 $stage-review-loop 轻量验收当前工作，并修复阻断性问题。
```

### 独立审查者模式

审查力度更强，适用于发布前的最终确认或高风险的变更。在该模式下，可用时优先使用独立 reviewer、新线程或新的模型上下文。

如果当前环境不支持完全的上下文隔离，它会明确告知用户：当前仅为“在现有上下文中模拟独立审查”，而非真正意义上的隔离环境审查。

示例指令：

```text
使用 $stage-review-loop 独立审查者模式，检查最终结果是否符合原始需求。
```

## 和 Codex 官方审查的关系

本 Skill 并非旨在替代 Codex 内置的代码审查（Code Review）或 PR 审查。对于代码 diff 级别的细微问题，官方的 Review 功能依然是首选。

本 Skill 的定位是任务级验收：关注宏观层面上的问题——任务是否真正达成了用户要求？是否进行了实际验证？是否存在未处理的阻断性异常？

推荐的最佳实践是将两者结合：先使用官方审查排查 diff 中的具体代码问题，再运行本 Skill 进行整体任务的验收把关。

## 边界与限制

本 Skill 能够有效减少验收遗漏、需求蔓延和无依据的收尾，但无法确保拦截所有问题。对于依赖未提供的外部事实、特定领域知识、缺乏测试覆盖、模糊需求，或必须依赖外部判断、用户决策或真正独立审查的场景，仍需人工或外部介入。

它的设计初衷是保持轻量。除非用户明确下达指令要求执行深度审查，否则它不应演变成一个繁重的发布流程。

## 安装

将 Skill 文件夹复制到 Codex skills 目录：

```text
stage-review-loop/
  SKILL.md
  agents/
    openai.yaml
```

然后按名称调用：

```text
使用 $stage-review-loop 验收当前工作。
```
